# AIコードレビュー依頼テンプレート

**バージョン**: 1.0.0
**作成日**: 2026-06-13

---

> このテンプレートは、Claude Code / ChatGPT / Copilot 等のAIにコードレビューを依頼するための
> プロンプトテンプレートです。コピーして使用してください。

---

## 使い方

1. 以下の「レビュー依頼プロンプト」をコピーする
2. `[コードをここに貼る]` の部分に実際のコードを貼る
3. `[プロジェクトの概要]` と `[使用ライブラリ]` を埋める
4. AIに送信する
5. 「レビュー結果の読み方」に従って結果を解釈する

```
## レビュー観点（全て確認してください）

以下の観点でレビューし、問題点を全て列挙してください。

### 1. セキュリティ
- APIキー・パスワードがコードに直書きされていないか
- .envを使って秘密情報を外部化しているか
- ログに秘密情報・個人情報が出力されていないか
- GitHubにpushしてはいけないファイルが.gitignoreに含まれているか

### 2. エラー処理
- `except: pass` や `catch {}` のような例外の握り潰しがないか
- 認証エラー・権限エラーで即時停止しているか
- ネットワークエラーにリトライ処理があるか
- エラー発生時にログに原因・IDが記録されているか
- 想定外の例外をキャッチしているか

### 3. ログ
- INFO / WARNING / ERROR が適切に使い分けられているか
- 処理件数（合計・成功・失敗・スキップ）がログに出るか
- 失敗したデータのIDがログに出るか
- ログに秘密情報が含まれていないか
- ログファイルへの保存が実装されているか

### 4. 重複・二重実行防止
- 同じデータを二重処理しないか（処理済みIDの管理）
- Cron実行の場合、二重起動防止（ロックファイル等）があるか

### 5. dry-run対応
- `--dry-run` フラグが実装されているか
- dry-run時は出力・変更が発生しないか

### 6. 保守性・可読性
- 関数が1つの責務だけを持っているか（長すぎる関数がないか）
- 変数名・関数名が意図を表しているか
- マジックナンバー・マジック文字列が定数化されているか
- 処理フローを把握するのに必要なコメントがあるか

### 7. 重複コード
- 同じ処理が複数箇所にコピーされていないか
- 共通化・関数化できる箇所がないか

### 8. パフォーマンス
- 不必要なAPI呼び出しを繰り返していないか
- 大量データで遅くなる処理がないか（N+1問題等）
- タイムアウト設定があるか

### 9. 運用性
- 環境変数の説明（.env.example）が書かれているか
- READMEに起動方法・停止方法が書かれているか
- 手動で失敗データを再実行できる仕組みがあるか

### 10. 横展開性
- 業務固有の設定が設定ファイルに外出しされているか
- 固有処理と汎用処理が分離されているか
- 設定を変えるだけで別の案件に転用できるか

## 出力形式

問題点は以下の形式で出力してください。
優先度の高い順（重大 → 高 → 中 → 低）に並べてください。

---

### 問題No.{番号}

**重大度**: 重大 / 高 / 中 / 低
**観点**: セキュリティ / エラー処理 / ログ / 重複防止 / dry-run / 保守性 / 重複コード / パフォーマンス / 運用性 / 横展開性
**該当箇所**: ファイル名:行番号 または 関数名
**問題点**: （何が問題か）
**理由**: （なぜ問題か。どんなリスクがあるか）
**修正案**: （どう直せばよいか。コード例があれば示す）
**優先度**: 本番前に必須 / 次回対応 / 改善推奨

---

最後に以下もまとめてください：
- 良い点（そのまま残すべき実装）
- 総合評価（本番投入可 / 修正後に投入 / 再設計が必要）
- 修正が必須な項目の数
```

## レビュー依頼プロンプト（簡易版）

時間がない場合や、部分的なコードのレビューには簡易版を使用してください。

```
以下のPythonコードをレビューしてください。
特に以下の観点で問題点を洗い出してください。

確認観点:
- セキュリティ（APIキー直書き・ログへの秘密情報出力）
- エラー処理（例外の握り潰し・認証エラーの扱い）
- ログ（件数出力・失敗原因の記録）
- 重複防止（二重処理・二重起動）

出力形式:
- 重大度（重大/高/中/低）
- 問題点
- 修正案

コード:
[コードをここに貼る]
```

## レビュー結果の読み方・対応方針

### 重大度の定義

| 重大度 | 意味 | 対応 |
|---|---|---|
| **重大** | セキュリティリスク・データ破損・処理不能が発生する | 本番投入禁止。必ず修正する |
| **高** | 障害時に追跡・復旧できなくなる問題 | 本番前に必ず修正する |
| **中** | 品質が下がるが緊急ではない問題 | 次回スプリントで対応 |
| **低** | リファクタリング・改善推奨 | 時間があれば対応 |

### 本番投入判断

| 状況 | 判断 |
|---|---|
| 重大・高が0件 | **本番投入OK** |
| 重大が1件以上 | **本番投入禁止** |
| 高が1〜2件 | 修正してから投入 |
| 中・低のみ | 投入後に対応でも可 |

## セルフチェック（AI依頼前に確認）

AIに依頼する前に、自分でも以下を確認してください。

```
[ ] APIキーがコードに含まれていない
[ ] .envを使っている
[ ] except: pass がない
[ ] ERRORログが出力される
[ ] 処理件数がログに出る
[ ] dry-runモードがある
[ ] ロックファイル（Cron実行の場合）がある
```

## 活用例

### ケース1：開発完了後のレビュー
→ フル版プロンプトを使って全観点でレビューを依頼する

### ケース2：急いで動作確認したい
→ 簡易版プロンプトでセキュリティ・エラー処理だけ確認する

### ケース3：特定の観点だけ確認したい
→ フル版プロンプトの不要な観点を削除して依頼する
