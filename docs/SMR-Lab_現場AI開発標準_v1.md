# SMR-Lab 現場AI開発標準 v1

**バージョン**: 1.0.0
**作成日**: 2026-06-13
**作成者**: SMR-Lab
**対象**: 現場業務改善ツール全般（BOT・API連携・GAS・CSV整形・Cron処理等）

---

## 1. この標準の目的

この標準は、SMR-Labが開発・運用するすべての現場AIツールに共通して適用するルールとテンプレートをまとめたものです。

### 1.1 なぜ標準が必要か

現場でAIを使った業務改善ツールを作ることは、今や技術的なハードルではありません。  
問題は「作れること」と「安全に運用し続けられること」は全く別の話である点です。

| よくある失敗 | 結果 |
|---|---|
| APIキーをコードに直書き | GitHubに漏洩 |
| エラー処理なし | 失敗しても気づかない |
| ログなし | 何が起きたか追跡不能 |
| 要件定義なし | 何を作るか途中でぶれる |
| テストなし | 本番で初めて不具合発覚 |
| 運用手順なし | 担当者変更で誰も動かせない |
| 障害対応なし | 止まったまま放置 |

この標準は、上記の失敗を防ぐために必要な設計・実装・運用のチェックポイントを定義します。

### 1.2 この標準が対象とする案件

- Chatwork / LINE / Slack 収集・通知BOT
- 入荷通知・受注通知BOT
- メール振り分けBOT
- Threads / Instagram 自動投稿
- Shopify / ネクストエンジン連携
- Google Apps Script (GAS) 自動化
- CSV整形・データ変換処理
- API連携処理
- 社内定期バッチ処理
- Cron / タスクスケジューラ実行

### 1.3 この標準が対象外とするもの

- 機械学習モデルの学習・チューニング
- インフラ構築（サーバー・ネットワーク設計）
- アプリケーション開発（スマホアプリ・Webサービス等）

## 2. AI開発で失敗する典型パターン

### パターン1：「動いた」で終わり
動作確認しただけで本番投入。エラー時に何も記録されず、失敗に気づくのが翌日以降。

### パターン2：要件が口頭のまま
「Chatworkのメッセージを集めたい」という曖昧な依頼でコードを書き始める。途中で「そういう意味じゃない」が多発。

### パターン3：APIキーをGitHubに上げる
.env を使わずにコードに直書き。GitHubにpushした瞬間に漏洩。最悪の場合、第三者に課金される。

### パターン4：エラーをcatchして握り潰す
`except: pass` や `try { } catch { }` で例外を無視。ツールは動いているように見えるが、実は毎回失敗している。

### パターン5：ログが何もない
何件処理したか、いつ実行したか、何が失敗したかが一切わからない。

### パターン6：二重実行を考慮していない
Cronで起動しているのに前の処理が終わっていない。同じデータを二重処理して重複が発生。

### パターン7：担当者が変わって誰も動かせない
READMEがない、環境変数の説明がない、起動方法が口頭伝承のみ。

### パターン8：本番データを直接加工して壊す
バックアップなしでCSVを上書き。元データが失われる。

## 3. 現場AI開発の基本原則

### 原則1：目的を先に書く
コードより先に「何のために作るのか」を1行で書く。それができなければ開発しない。

### 原則2：秘密情報はコードに書かない
APIキー・パスワード・トークンは `.env` に書く。コードには変数名だけ。

### 原則3：必ずログを残す
何件処理したか、いつ実行したか、何が失敗したかを `logs/` に記録する。

### 原則4：エラーは握り潰さない
例外はcatchしても必ずログに出力する。原因が追跡できなければ修正もできない。

### 原則5：本番データを直接触らない
処理前にバックアップを取る。dry-runモードで確認してから本番実行する。

### 原則6：人間が止められる設計にする
完全自動化でも、異常検知時は通知を出し、手動介入できる逃げ道を残す。

### 原則7：誰でも引き継げるREADMEを書く
「この人しか動かせない」を作らない。

### 原則8：横展開を意識して作る
次の案件でも使えるように、固有処理とテンプレート処理を分離する。

### 4.1 各フェーズの完了条件

| フェーズ | 完了条件 |

| 目的の明確化 | 1行で説明できる |

| 運用準備 | 第三者が運用手順書だけで起動・停止・ログ確認できる |
| 本番投入 | dry-run成功・バックアップ確認済み |

## 6. ログ標準

詳細は `docs/ログ設計標準.md` を参照。

### 6.1 必須ログ項目

| 項目 | 説明 |
|---|---|
| 実行日時 | ISO 8601形式（例: 2026-06-13T09:00:00+09:00） |
| 実行ID | UUID等。再実行時に区別できる |
| 処理件数 | 全件数・成功・失敗・スキップ |
| エラー内容 | エラーメッセージ・スタックトレース |
| 実行モード | dry-run / 本番 |

### 6.2 ログレベル

| レベル | 用途 |
|---|---|
| INFO | 正常処理・件数報告 |
| WARNING | 想定範囲内の軽微な問題 |
| ERROR | 処理失敗・例外発生 |
| CRITICAL | システム停止レベルの障害 |

## 7. エラー処理標準

詳細は `docs/エラー処理標準.md` を参照。

### 7.1 基本方針

- エラーを握り潰さない（`except: pass` 禁止）
- エラーはログに記録してから処理継続 or 停止を判断する
- 続行可能なエラー（1件スキップ）と停止必須なエラー（認証失敗）を区別する

### 7.2 エラー分類

| 分類 | 対応 |
|---|---|
| 認証エラー | 即時停止・通知 |
| APIレートリミット | リトライ（指数バックオフ）|
| データ不正（1件） | スキップしてログ記録 |
| ファイル不存在 | 即時停止 |
| ネットワークエラー | リトライ3回→停止 |

## 8. 重複防止標準

### 8.1 重複の種類

| 種類 | 原因 | 対策 |
|---|---|---|
| 二重実行 | Cronが重なる | PIDファイル / ロックファイル |
| データ重複 | 同じデータを二回処理 | 処理済みIDを記録して除外 |
| API二重送信 | リトライ時に同じリクエストを送る | 冪等キー（Idempotency Key） |

### 8.2 PIDファイルによる二重起動防止（Python例）

```python
import os, sys

LOCK_FILE = "run.lock"

if os.path.exists(LOCK_FILE):
    print("既に実行中です。終了します。")
    sys.exit(1)

with open(LOCK_FILE, "w") as f:
    f.write(str(os.getpid()))

try:
    # 処理
    pass
finally:
    os.remove(LOCK_FILE)
```

## 9. テスト標準

詳細は `docs/テスト項目テンプレート.md` を参照。

### 9.1 テストの順序

1. dry-run（実際の変更なし）
2. 小規模データで正常系
3. 異常系（空データ・不正データ・重複）
4. API失敗・タイムアウト模擬
5. 二重起動テスト
6. 本番環境と同等のデータで全量テスト

### 9.2 合格条件

- 全テスト項目に結果が記載されている
- ERRORログが出ていないこと
- 処理件数（入力 = 成功 + スキップ + 失敗）が一致すること
- dry-runで実際の変更が発生していないこと

## 11. 障害対応標準

詳細は `docs/障害対応手順書テンプレート.md` を参照。

### 11.1 障害発生時の初動（5分以内）

1. ログを確認して「何が失敗したか」を把握する
2. 影響範囲（件数・期間）を確認する
3. 処理が止まっているなら手動モードに切り替える
4. 必要なら関係者に連絡する

### 11.2 障害レベル分類

| レベル | 定義 | 対応目標 |
|---|---|---|
| L1 | 業務完全停止 | 30分以内に暫定復旧 |
| L2 | 一部機能停止 | 2時間以内に対応 |
| L3 | 軽微な不具合 | 翌営業日以内に対応 |

## 12. 横展開標準

詳細は `docs/横展開判断シート.md` を参照。

### 12.1 横展開しやすい設計

- 設定値（URL・シートID・メールアドレス等）は config.json または .env に外出し
- 業務固有の処理は関数・クラスとして分離
- READMEだけで別担当者が動かせる
- 処理フローのドキュメントがある

### 12.2 横展開の活用先

- 他部署・他プロジェクトでの再利用
- SMR-Labのnote記事・Kindle書籍のネタ
- 顧客向け相談・支援の提案材料

## 13. Claude Code / Codex / ChatGPT への依頼方法

### 13.1 依頼前に準備すること

AIにコードを書かせる前に、以下を自分で決めること。これが曖昧なままだと、AIが作るものも曖昧になる。

```
[ ] 何を自動化するか（1文で説明できる）
[ ] 入力データの形式（CSV・API・Spreadsheet等）
[ ] 出力データの形式と保存先
[ ] エラー時にどうするか
[ ] ログをどこに保存するか
[ ] dry-runモードが必要か
[ ] 使う言語・ライブラリ
```

### 13.2 良い依頼文の例

```
以下の仕様でPythonスクリプトを作ってください。

目的: ネクストエンジンAPIから当日の受注データを取得してGoogleスプレッドシートに書き込む

入力: ネクストエンジンAPI（エンドポイント・認証方法は.envで管理）
出力: Googleスプレッドシート（SHEET_IDは.envで管理）、シート名「受注_YYYYMMDD」

エラー処理:
- API認証エラーは即時停止してERRORログを出す
- 1件のデータ不正はスキップしてWARNINGログを出す
- ネットワークエラーは3回リトライ

ログ: logs/YYYYMMDD.log に INFO/WARNING/ERROR で出力
dry-run: --dry-run フラグで実際の書き込みをスキップ
重複防止: 実行IDをスプレッドシートの「実行ログ」シートに記録

ディレクトリ構成:
project/
├─ src/
├─ logs/
├─ .env
├─ .env.example
└─ README.md
```
