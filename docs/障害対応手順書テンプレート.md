## 障害レベル定義

| レベル | 定義 | 対応目標時間 | 例 |
|---|---|---|---|
| **L1 緊急** | 業務が完全に止まっている | 30分以内に暫定復旧 | 全件処理失敗・APIキー漏洩 |
| **L2 重大** | 一部機能が停止・データ欠損の可能性 | 2時間以内に対応開始 | 一部データが取得できない |
| **L3 軽微** | 影響は小さいが修正が必要 | 翌営業日以内に対応 | ログにWARNINGが多い |

## 1. 障害発生時の初動（5分以内に実施）

```
Step 1: 何が起きているかを把握する（ログを見る）
  ↓
Step 2: 影響範囲を確認する（何件・いつから？）
  ↓
Step 3: 処理を一時停止する（被害を広げない）
  ↓
Step 4: 関係者に第一報を入れる
```

### Step 1: ログを確認する

```bash
# 今日のエラーを確認
grep "ERROR\|CRITICAL" /home/user/project-name/logs/$(date +%Y%m%d).log

# 直近10件のエラー
grep "ERROR\|CRITICAL" /home/user/project-name/logs/*.log | tail -n 10
```

### Step 2: Cronが動いているか確認する

```bash
# プロセスが動いているか
ps aux | grep main.py

# Cron実行ログを確認
cat /home/user/project-name/logs/cron.log
```

### Step 3: 自動実行を一時停止する

```bash
crontab -e
# 対象行の先頭に # を付けてコメントアウトし、自動実行を止める
# 0 9 * * * /home/user/project-name/run.sh
```

### Step 4: 第一報のテンプレート

```
【障害発生】{プロジェクト名}

発生日時: YYYY-MM-DD HH:MM
発見方法: 通知 / ログ確認 / 出力確認
症状: （例）本日分のデータがスプレッドシートに反映されていない
影響範囲: 現在調査中
対応状況: ログ確認中

詳細が判明次第、続報をお送りします。
```

## 2. 影響範囲確認

| 確認項目 | 確認方法 | 確認内容 |
|---|---|---|
| 未処理件数 | ログの「合計件数」と出力件数を比較 | 何件が処理されていないか |
| 発生開始日時 | ログの最初のERROR行を確認 | いつから失敗しているか |
| 最終成功日時 | ログでINFO「処理完了」を最後に確認 | 最後に正常終了したのはいつか |
| 出力先の状態 | スプレッドシート等を直接確認 | データが欠損しているか |
| バックアップの状態 | `ls backup/` | 直前のバックアップがあるか |

## 3. ログ確認（詳細）

### 3.1 エラー種別を特定する

```bash
# エラー内容の全量確認
grep "ERROR\|CRITICAL" /home/user/project-name/logs/$(date +%Y%m%d).log

# 特定のエラーを検索
grep "AuthenticationError\|認証" /home/user/project-name/logs/$(date +%Y%m%d).log
grep "ConnectionError\|Timeout" /home/user/project-name/logs/$(date +%Y%m%d).log
```

### 3.2 エラー種別と対応

| エラー文言 | 原因 | 対処 |
|---|---|---|
| `AuthenticationError` / `認証失敗` | APIキーが無効・期限切れ | APIキーを更新→Bitwardenも更新 |
| `ConnectionTimeout` / `Timeout` | ネットワーク問題 | 接続状況確認→リトライ |
| `PermissionError` / `権限エラー` | アクセス権なし | 権限を確認・修正 |
| `FileNotFoundError` | ファイルが見つからない | パスと存在を確認 |
| `RateLimitError` | API呼び出し上限 | 時間を置いて再実行 |
| `KeyError` / `ValueError` | データ形式の問題 | データを確認してスキップ設定 |

## 4. 最終成功日時の確認

```bash
# 最後に「処理完了」が出たのはいつか
grep "処理完了" /home/user/project-name/logs/*.log | tail -n 5
```

最終成功日時からの未処理分が「手動対応が必要な期間」です。

# 失敗件数を確認
wc -l /home/user/project-name/data/failed_ids.txt

## 6. 手動復旧

### 6.1 バックアップから元データを戻す

```bash
# バックアップを確認
ls /home/user/project-name/backup/

# 最新のバックアップから復元
cp /home/user/project-name/backup/{YYYYMMDD_HHMMSS}/output.csv \
   /home/user/project-name/data/output.csv
```

### 6.2 スプレッドシートの手動修正

1. 最終正常処理日以降のデータを確認する
2. 未反映分を元データ（Excelや管理システム）から手動で転記する
3. 転記後、修正した行に「手動修正」フラグを立てる

## 7. 再実行

### 7.1 エラー原因を修正してから再実行する

```bash
# 原因確認・修正後
# dry-runで確認
python src/main.py --dry-run

# 問題なければ本番実行
python src/main.py
```

### 7.2 失敗データだけ再実行する

```bash
# 失敗IDを指定
python src/main.py --retry-ids 123,456,789
```

### 7.3 欠損期間を再実行する（日付指定が可能な場合）

```bash
# 特定日付を再処理（仕様による）
python src/main.py --date 2026-06-12
```

### 7.4 再実行前のチェック

```
[ ] エラーの原因を特定・修正した
[ ] バックアップが取れている
[ ] dry-runで処理件数を確認した
[ ] 出力先を確認して二重書き込みが発生しないことを確認した
[ ] Cronの自動実行がコメントアウトされている（手動再実行中は止めておく）
```

## 9. 原因分類

障害の根本原因を以下から選択して記録する。

| 分類 | 例 |
|---|---|
| **認証・権限** | APIキー期限切れ、権限変更 |
| **外部サービス** | API仕様変更、外部システム障害 |
| **データ品質** | 想定外のデータ形式、空データ |
| **ネットワーク** | 接続タイムアウト、一時的な切断 |
| **設定ミス** | 環境変数の誤り、設定ファイルの誤り |
| **バグ** | コードの論理エラー |
| **リソース不足** | ディスク満杯、メモリ不足 |
| **不明** | 原因を特定できなかった |
